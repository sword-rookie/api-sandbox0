Here is a comprehensive breakdown of every single file in the `/frontend`, `/orchestrator`, and `/volumes` directories, explaining why they exist and how they fit into the Clarity Machine project.

---

## 1. The `/frontend` Directory
This contains the React/Next.js web application that the user interacts with in their browser.

### Application Pages (`app/` directory)
- **`app/page.tsx`**: The main landing page / home page of the website.
- **`app/layout.tsx`**: The global layout wrapper for the Next.js app (defines the HTML structure, global fonts, and wraps all pages).
- **`app/globals.css`**: Contains global CSS styles, Tailwind imports, and core design system variables (colors, fonts).
- **`app/favicon.ico`**: The small icon displayed in the browser tab.
- **`app/dashboard/page.tsx`**: The main user dashboard where they can see their sandboxes and orchestrator stats.
- **`app/login/page.tsx`**: The login page UI (accepts email/password or OAuth).
- **`app/signup/page.tsx`**: The registration page UI.
- **`app/forgot-password/page.tsx`**: The UI for users to request a password reset email.
- **`app/reset-password/page.tsx`**: The UI where users type their new password (using the token sent via email).
- **`app/profile/[username]/page.tsx`**: A dynamic route that displays a specific user's public profile based on their username.
- **`app/settings/account/page.tsx`**: The UI where a logged-in user can manage their account settings (bio, location, etc.).
- **`app/privacy/page.tsx` & `app/terms/page.tsx`**: Static legal pages (Privacy Policy and Terms of Service).

### Configuration & Tooling Files
- **`proxy.ts`**: A custom script likely used to proxy API requests during development to bypass CORS issues or route traffic to the Orchestrator.
- **`next.config.ts`**: The configuration file for Next.js (defines routing rules, environment variable mappings, image domains).
- **`tsconfig.json`**: TypeScript configuration, dictating strictness rules and path aliases for compiling the frontend code.
- **`package.json`**: Defines the project metadata, run scripts (like `npm run dev`), and lists all the npm dependencies.
- **`package-lock.json`**: An auto-generated file that locks down the exact versions of dependencies installed to ensure consistency across machines.
- **`postcss.config.mjs`**: Configuration for PostCSS (used by TailwindCSS to process modern CSS).
- **`eslint.config.mjs`**: Linter configuration to enforce coding standards and catch errors before compiling.
- **`.gitignore`**: Tells Git which files to ignore (like `node_modules` or `.next`).
- **`next-env.d.ts`**: Auto-generated TypeScript declarations for Next.js specific types.
- **`README.md` & `AGENTS.md` & `CLAUDE.md`**: Markdown documentation files explaining how to run the frontend and providing context for AI agents.

## 2. The `/orchestrator` Directory
This is the Go (Golang) backend service. It acts as the "brain" (Clarity Machine), handling user authentication, database interactions, and (eventually) talking to the Docker daemon to build sandboxes.

### Core Entry Point & Build Files
- **`cmd/main.go`**: The entry point of the Go backend. It loads the config, initializes the server, and starts listening on port `8081`.
- **`main`**: The compiled Linux executable binary of the Go application (generated when you run `go build`).
- **`go.mod`**: Defines the Go module path and lists direct dependencies (like GORM, Gorilla Mux, UUID).
- **`go.sum`**: Checksums of the dependencies to ensure they haven't been tampered with.
- **`Dockerfile`**: Defines how to package the orchestrator into a container (though we've decided to run it natively instead for better access to Docker).

### Database Migrations (`migrations/` directory)
- **`000001_init_schema.up.sql`**: The raw SQL commands to create the `users`, `refresh_tokens`, `password_reset_tokens`, and `audit_logs` tables in PostgreSQL.
- **`000001_init_schema.down.sql`**: The raw SQL to drop those tables if we need to rollback the migration.

### Internal Logic (`internal/` directory)
- **`server/server.go`**: Sets up the HTTP server, initializes the database repository, registers all the API routes, and sets up CORS middlewares.
- **`config/config.go`**: Reads environment variables (from `.env`) and parses them into a structured Go object used throughout the app.

#### Authentication (`auth/` directory)
- **`handler.go`**: Contains the HTTP endpoint functions (like `Login`, `Register`, `ForgotPassword`). It parses JSON requests, calls the service layer, and returns HTTP responses.
- **`service.go`**: Contains the core business logic (hashing passwords, validating tokens, returning errors).
- **`tokens.go`**: Logic for generating secure JWTs and Refresh Tokens.
- **`refresh.go`**: Logic for validating expired JWTs and issuing new ones using valid refresh tokens.
- **`oauth.go`**: Handles third-party logins (like GitHub/Google OAuth).

#### Database Integration (`repository/` directory)
- **`repository.go`**: Defines a strict Interface (`Repository`) that outlines every possible database action (like `CreateUser` or `UpdatePassword`). This keeps the app modular.
- **`postgres.go`**: The concrete implementation of the Repository interface. It uses GORM and raw SQL strings to actually read/write data to PostgreSQL.

#### Data Models (`models/` & `dto/`)
- **`models/user.go`, `session.go`, `audit.go`, `password_reset.go`**: These define the Go `structs` that map directly to rows in the PostgreSQL tables.
- **`dto/auth.go`, `profile.go`**: DTO stands for Data Transfer Object. These are structures defining the exact JSON shape expected in incoming HTTP requests or outgoing API responses.

#### Security & Builder (`security/` & `builder/`)
- **`security/argon2.go`**: Uses the highly secure Argon2 algorithm to hash user passwords so they are never stored in plaintext.
- **`builder/builder.go`**: (Currently a skeleton) This is where the logic will go to take a user's GitHub repository and use the host's Docker socket to build it into an isolated Sandbox container.

#### Empty Architectural Folders (`ai/`, `graph/`, `proxy/`, `sandbox/`)
These folders exist to define the future architecture. They will house the LLM integration logic, the node-based visual workflow system, the reverse proxy router for sandboxes, and the sandbox lifecycle manager, respectively.

## 3. The `/volumes` Directory
This directory is mapped as a volume mount to persist data on the host machine.
- **`clarity.db`**: This is a leftover artifact from when the Orchestrator was using SQLite as its local database. Since we successfully migrated to PostgreSQL in Docker, this file is now obsolete and can safely be deleted.
